# Shinkansen 自動化測試流程說明

> 這份文件是寫給 **Jimmy** 看的，不是寫給 Claude 看的。
>
> 目的：不小心關掉 Claude 或終端機後，重新打開也能照著步驟繼續工作。不需要記任何技術細節。

---

## 這是什麼？

我們花了一段時間幫 Shinkansen 這個 Chrome Extension 建立了一套**自動化回歸測試系統**。它的核心價值是：當你（或 Cowork）改了 Extension 的偵測 / 序列化 / 注入邏輯（例如修 bug、加新功能），不需要人工去打開 Gmail / Wikipedia / Twitter 一個一個重測，電腦會在 ~15 秒內跑過 11 條測試，告訴你「歷史踩過的 bug 有沒有再炸」。

**目前 (v0.59) 的測試規模**：
- **10 條 regression spec**：對應 v0.49–v0.58 期間踩過的每一個 bug，每條都用 synthetic HTML fixture (不依賴 live 網站) 加 canned 翻譯回應，秒級完成。
- **1 條 version-check spec**：強制你 bump 版本號時必須同步更新測試常數，是個「不更新就 fail」的 forcing function。

每條 regression spec 都做過 sanity check (把對應的 fix 拆掉,確認測試會精準炸),所以可以信賴它「真的會抓到 bug」。

這套系統分成兩個工作環境，各司其職：

**Cowork（就是你現在在跟 Claude 講話的這個視窗）**：負責改 Extension 的程式碼、更新 SPEC 文件、跟你討論設計方向。

**Claude Code（終端機裡的另一個 Claude）**：負責跑自動化測試、比對結果、做 git 相關操作。它是終端機工具，在你的電腦上直接執行程式。

你不需要記住哪件事歸哪邊——你只要「想改什麼就跟 Cowork 講」「想跑測試就跟 Claude Code 講」。兩邊的 Claude 都讀同一份專案文件，上下文是相通的。

## 開始一個新的工作階段

### 如果你想改 Extension 或討論設計

打開 Claude 桌面版，確認左上角的 Cowork 模式選到「Immersive Translation Clone」這個資料夾。這就是目前你看到的這個視窗。直接跟我講你想做什麼就好，例如：

- 「Wikipedia 的 Tokyo 條目有個新的維護模板叫 XXX，幫我加進排除清單」
- 「References 區的問題我想再試一次，這次從 YYY 方向查」
- 「SPEC.md 第 5 節我想加一條規則」

我會處理改 code、改文件、bump 版本號、備份、告訴你下一步該做什麼。

### 如果你想跑自動化測試

**步驟 1：打開終端機**

按 `⌘ + 空白鍵` 叫出 Spotlight，輸入「終端機」按 Enter。

**步驟 2：進到專案資料夾**

在終端機輸入 `cd ` （cd 後面有一個空格），然後從 Finder 把「Immersive Translation Clone」資料夾直接拖進終端機視窗。終端機會自動填入完整路徑。按 Enter。

**步驟 3：啟動 Claude Code**

在終端機輸入：

```
claude
```

按 Enter。如果 Claude Code 問你要不要延續上次的對話，選「是」可以看到先前的進度；選「否」會開新對話（開新對話會自動讀 CLAUDE.md，知道專案背景）。

## 常用指令：直接複製貼上給 Claude Code

這些是你最常會用到的指令，直接整段貼進 Claude Code 的對話框按 Enter 就行。

### 指令 A：跑全部回歸測試（最常用）

```
跑 npm test
```

就這樣，一行。會跑全部 11 條測試 (10 regression + 1 version-check)，~15 秒結束。
全綠 = 改動沒踩到任何歷史 bug；有紅 = Claude Code 會把哪一條哪一行斷言炸了給你看。

用途：Cowork 剛改完 code 後，你想確認改動沒有副作用。或是隨時想跑一次健康檢查。

### 指令 B：只跑回歸測試（不含 version-check）

```
跑 npm run test:regression
```

用途：你只想驗證 inject / serialize / detect 路徑沒壞，不想被 version drift check 打擾。

### 指令 C：跑單一測試 (debug 用)

```
跑 npx playwright test test/regression/inject-gmail-button.spec.js
```

把檔名換成你想跑的那條 spec。spec 檔名都在 `test/regression/` 底下，例如:
- `inject-gmail-button.spec.js` (Gmail 按鈕巢狀 a)
- `inject-wiki-ambox.spec.js` (Wikipedia 維護模板)
- `inject-br-paragraph.spec.js` (MJML 用 br 當段落)
- `pure-roundtrip.spec.js` (序列化健康檢查)
- ...等

用途：某條測試紅了,你想單獨跑那條看詳細失敗訊息,不想等其他 10 條跑完。

### 指令 D：新增一條 regression 測試

```
我在 XX 網站 / XX 結構發現一個 bug,Cowork 已經修好了。
請參考 test/regression/inject-gmail-button.spec.js 的格式,建一條新的 regression spec
鎖死這個 bug。fixture HTML 走 synthetic (不要依賴 live 網站)。
建好後跑一次 npm test 確認綠燈,然後做 sanity check
(把對應的 fix 暫時拆掉,確認測試會精準炸,再 restore)。
```

用途：踩到新 bug 之後,把它變成永久回歸測試,以後不會再踩同一個。

### 指令 E：檢查 git 狀態

```
跑 git log --oneline -10 && git tag,告訴我目前有哪些版本和 tag。
```

用途：搞不清楚目前是哪個版本、有沒有還沒 commit 的變動時用。

### 指令 F：回復到某個舊版本

```
把 shinkansen/ 回復到 v0.XX(把 XX 換成實際版本號,例如 0.28)。
先 stash 目前的變動避免遺失,然後 git checkout v0.XX -- shinkansen/ 還原。
完成後告訴我要怎麼 reload extension。
```

用途：新版本改壞了想回到可用狀態時用。

## 標準工作循環

一個完整的改動通常會是這樣的流程，你照著跑就對了：

**第 1 步** — 在 **Cowork** 跟我講你想改什麼。我會處理改 code、改 SPEC、bump 版本號、commit、tag。你不用動手。

**第 2 步** — 打開終端機進專案資料夾，啟動 Claude Code。

**第 3 步** — 貼**指令 A**（`跑 npm test`）。Claude Code 會自動跑全部 11 條測試，~15 秒後告訴你結果。

**第 4 步** — 看結果，只有兩種可能：

- **全綠 (11 passed)**：太好了，這次改動沒踩到任何歷史 bug。你可以收工，或繼續下一個改動。
- **有紅 (X passed, Y failed)**：Claude Code 會把哪一條測試、哪一行斷言、實際值 vs 期望值都印出來。回 Cowork 把整段失敗訊息貼給我，我會根據失敗訊息判斷是改錯還是測試該更新。

**第 5 步**（可選）— 到你的 Chrome 實際載入最新版 Extension 手動驗證。打開 `chrome://extensions/`，找到 Shinkansen，按「重新載入」按鈕。然後到目標網頁按 `Option + S` 觸發翻譯。

## 怎麼讀測試結果

新版的測試是「全綠 / 有紅」的 pass/fail 模式,不再產 JSON 報告需要人工解讀。
Claude Code 跑完後會直接給你一個總結,看起來像這樣:

```
Running 11 tests using 1 worker
  ✓   1 test/regression/detect-stratechery.spec.js   (1.3s)
  ✓   2 test/regression/detect-twitter-widget.spec.js (1.3s)
  ✓   3 test/regression/inject-br-paragraph.spec.js   (1.3s)
  ...
  ✓  11 test/version-check.spec.js                    (1.2s)

  11 passed (15.2s)
```

**全綠時你只需要看「11 passed」**——代表 v0.49–v0.58 期間踩過的所有 bug
都沒有再炸,改動安全。

**有紅時** Claude Code 會把失敗訊息整段印出,通常長這樣:

```
  ✘  3 test/regression/inject-gmail-button.spec.js (2.1s)

    Error: 注入後 td 直接子 anchor 應為 1,實際 2
    expect(received).toBe(expected) // Object.is equality
    Expected: 1
    Received: 2
       at test/regression/inject-gmail-button.spec.js:67:25
```

把這整段貼回 Cowork,我會告訴你是改錯了 (該回頭改 code) 還是測試需要更新
(該調整斷言)。**不要自己嘗試解讀斷言訊息**——這是工程師才需要懂的細節,
你的工作只是「把訊息傳回來」。

| Spec 檔名 | 對應 bug |

| `inject-gmail-button.spec.js` | Gmail 按鈕被翻成巢狀 a (v0.56/v0.58) |

## 這套工具**不會**幫你做的事

為了不踩到地雷，先說清楚它的限制：

**不會判斷翻譯品質**：譯文讀起來自然不自然、術語對不對，這類事情需要人讀過才知道。自動化測試只能驗證「哪些段落被抓進去翻」，不能驗證「翻得好不好」。

**不會真的呼叫 Gemini 翻譯**：測試是唯讀的，只會問 Extension「你看到什麼」，不會觸發真實翻譯，所以不會燒 API 額度，也不會把你的 API key 用掉。

**不會抓到翻譯過程中的 runtime error**：如果翻譯到一半 crash 了，測試看不到。要發現這類問題只能手動按 `Option + S` 實際翻譯一次。

**不會自動觸發 reload extension**：測試跑在一個乾淨的 Chromium，跟你日常用的 Chrome 是兩個不同的瀏覽器。如果你要在日常 Chrome 上驗證，還是要手動到 `chrome://extensions/` 按重新載入。

## 遇到狀況時的對照表

**狀況：Claude Code 說「找不到 node 指令」或「找不到 npm」**

表示 Node.js 沒裝或環境變數沒抓到。跟 Claude Code 講「我的 Node 可能沒裝好，幫我確認 `node -v` 跟 `npm -v` 有沒有輸出」。它會帶你排查。

**狀況：測試跑到一半卡住，或 Chromium 視窗打開後沒反應**

讓它跑滿 2 分鐘再說。如果還是卡住，按 `Control + C` 中斷，然後跟 Claude Code 講「剛才測試卡住，重跑一次」。

**狀況：git status 出現「Unable to create lock」或「index.lock」的錯誤**

在終端機跑 `ls .git/*.lock` 看看有沒有殘留 lock 檔案，有的話跑 `rm .git/*.lock` 刪掉再試。這個問題根因是 VS Code 的 git extension 在背景 polling，可以在 VS Code 設定裡搜尋「Git: Autorefresh」關掉就能避免。

**狀況：測試紅了,但失敗訊息看不懂**

正常的。把失敗訊息整段(從 `Running N tests` 到 `N failed` 為止)複製貼回
Cowork,我會解讀。不要自己嘗試改測試或改 code。

**狀況：version-check.spec.js 紅了**

代表 manifest 版本號 bump 過了但 `test/version-check.spec.js` 的
`EXPECTED_VERSION` 常數沒同步更新。這是設計成這樣的「forcing function」,
回 Cowork 跟我講「version-check 紅了,版本同步沒做完」即可。

**狀況：你忘記現在是第幾版、做到哪了**

兩個方法擇一：
1. 到 Cowork 跟我講「幫我看一下 manifest.json 和 SPEC.md 現在的版本號」
2. 到 Claude Code 貼**指令 D**

**狀況：重開電腦後想繼續之前的對話**

Cowork：打開 Claude 桌面版，左側應該看得到最近的對話，點進去就能接續。

Claude Code：在終端機進入專案資料夾，輸入 `claude` 啟動，它會問你要不要延續上次的 session，選「是」。

## 名詞速查

這些詞你不一定要會講，但看到知道大概是什麼就好。

**content.js**：Extension 的主要程式檔，負責偵測網頁段落、呼叫翻譯、替換文字。改翻譯邏輯都是改這個檔案。

**manifest.json**：Extension 的設定檔，裡面有版本號（例如 "version": "0.29"）。每次改 code 都要 bump 這個版本號。

**SPEC.md**：Extension 的規格文件。每次改行為都要同步更新這個檔案。

**CLAUDE.md**：給 Claude 看的協作規則文件。裡面寫了你的偏好、硬規則、工作風格。兩邊的 Claude 每次新對話都會先讀這份。

**git tag**：某個版本的「書籤」。例如 `v0.29` 這個 tag 對應當時 commit 的完整狀態，之後可以用它回復。

**Playwright**：一套自動化瀏覽器測試工具。我們用它啟動一個乾淨的 Chromium 載入 Extension、跑測試、產報告。

**debug API**（`window.__shinkansen`）：content.js 提供的唯讀接口，測試工具用它查詢「Extension 目前偵測到哪些單位」。永遠開著，不會影響正常翻譯。

**偵測單位 / 翻譯單位**：Extension 眼中的「一段要翻譯的文字」。例如一個 `<p>`、一個 `<li>` 通常各算一個單位。

**fixture**：自動化測試裡某個「測試場景」的設定。有兩種:
- 「啟動 Chromium 並載入 Shinkansen」這件事是一個 Playwright fixture
- `test/regression/fixtures/*.html` 是 synthetic 的網頁範本檔案,
  每條 regression spec 對應一份,長得像當年踩 bug 的真實網頁的縮小版

**regression test (回歸測試)**：把每個歷史 bug 都寫成一條測試,確保未來
的改動不會讓同一個 bug 再炸。Shinkansen 目前有 10 條,在 `test/regression/`。

**sanity check**：寫完一條 regression 測試後,刻意把對應的 fix 暫時拆掉,
確認測試會「精準炸在那一行斷言」。沒做過 sanity check 的測試不可信——
可能它根本不會因為 bug 復發而失敗。

**isolated world**：Chrome Extension 的一個技術隔離層，content.js 跑在這裡，跟網頁本身的 JavaScript 不會互相干擾。測試要看 `window.__shinkansen` 必須穿透到這層。

## 最後提醒

你不是工程師也不需要變成工程師。這份文件最重要的部分其實只有兩段：

1. **「標準工作循環」**那節——這是 90% 的日常用法
2. **「常用指令」**那節——想用的時候複製貼上

其他內容當字典查就好。任何時候卡住，回 Cowork 跟我描述狀況，我會幫你判斷下一步。

這套工具的目的是把你從「盯著瀏覽器一筆一筆看」解放出來，讓你專注在「要改什麼、為什麼改」而不是「怎麼改」。
